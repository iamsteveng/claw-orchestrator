## Codebase Patterns
- shared-config: schemas in `schemas.ts`, per-service config files (control-plane.ts etc.) import schemas and call `parse(process.env)` at module level; vitest-setup.ts sets required env vars so tests can import these modules
- Root vitest.config.ts `setupFiles` includes `packages/shared-config/src/vitest-setup.ts` — applies to ALL test files in the monorepo
- docker-client uses `"type": "module"` (ESM) to support execa v9 (ESM-only); other packages are CJS
- ExecaError in execa v9 has no public constructor — in tests, cast via `ExecaError as unknown as new (...) => ExecaError` for instantiation
- pnpm workspaces used: apps/* and packages/*; use `workspace:*` protocol for cross-package deps
- TypeScript module system: `module: Node16`, `moduleResolution: Node16`, `target: ES2023` (required for Node.js 22)
- Each package/app has its own tsconfig.json extending ../../tsconfig.json with rootDir/outDir overrides
- Build command per package: `tsc -p tsconfig.json`; typecheck: `tsc -p tsconfig.json --noEmit`
- Root vitest.config.ts covers all *.test.ts in packages/*/src/** and apps/*/src/**
- `pnpm.onlyBuiltDependencies: ["esbuild"]` required in root package.json to allow esbuild (vitest dep) post-install scripts in pnpm v10
- Path aliases in root tsconfig.json: @claw/shared-types -> packages/shared-types/src/index.ts, etc. (for IDE support)
- Prisma 5.22.0 installed at root; schema at prisma/schema.prisma; @prisma/client and prisma in root package.json; `@prisma/engines` and `@prisma/client` in pnpm.onlyBuiltDependencies
- Run `pnpm prisma generate` from root; `DATABASE_URL=file:./dev.db pnpm prisma migrate dev --name <name>` for migrations
- SQLite WAL mode enabled via `PRAGMA journal_mode=WAL;` at top of initial migration SQL file
- *.db, *.db-shm, *.db-wal in .gitignore
- buildDockerRunOptions readOnlyBindMounts: entries are hostPath:containerPath strings (no :ro); docker-client appends :ro at run time; both auth-profiles.json and .credentials.json are bind-mounted read-only

  - `prd.json` (all stories marked passes: true)
- **Learnings for future iterations:**
  - **workspace dir must be chmod 0o777**: The container's agent user (uid 1001) needs write access to `/workspace`. The health server (`checkWritable('/workspace')`) will return `workspace_mounted: false` if it's not writable, causing the health endpoint to return 503. Always call `chmod(dir, 0o777)` on host dirs before mounting them as container volumes.
  - **entrypoint chowns /home/agent to agent:agent**: After the container runs and is stopped, files in the home volume dir are owned by uid 1001. Cleanup must fix permissions first: `docker run --rm -v "${testDir}:/mnt" alpine chmod -R 777 /mnt` before `rm -rf`.
  - **Health check requires openclaw to be running**: `allOk = checks.openclaw && checks.workspace_mounted && checks.home_mounted`. The container starts openclaw gateway in the foreground — if it starts successfully (credentials valid), health returns 200. If credentials are invalid or openclaw fails, health stays 503.
  - **docker port output format**: `docker port name 3101` returns `"0.0.0.0:32768\n[::]:32768\n"` — use `.trim().match(/:(\d+)$/)` to extract port from the last line.
  - **Real auth files from host homedir work in CI**: `/home/ubuntu/.openclaw/agents/main/agent/auth-profiles.json` and `/home/ubuntu/.claude/.credentials.json` exist on this machine. Always check with `access()` first and create minimal fixtures only if missing.
  - **Container health returns 503 until openclaw starts**: Poll with 500ms intervals. The container typically reaches healthy state within 2-3 seconds.
