<learnings>
1. **Failure trigger is `seedWorkspace`, not `docker.run`**: The `/provision` endpoint creates dirs and seeds workspace only — docker.run is called in `/start`. Mock `seedWorkspace` to trigger provision failure.

2. **Retry behavior gap**: The provision endpoint returns 200 with FAILED status for an existing FAILED tenant with attempts < 3 (no re-provisioning occurs). The SPEC says FAILED should be treated like NEW in the relay's gate, but the implementation just returns the existing record. Tests for "retry increments attempts" need DB manipulation rather than repeated API calls.

3. **3-attempt cap check**: `existing.status === FAILED && provision_attempts >= 3` → 409. Direct DB update to set attempts=3 is the cleanest way to verify this cap in integration tests.

4. **provision_attempts starts at 0**: Created with `provision_attempts: 0`, then `{ increment: 1 }` on rollback → ends at 1 after first failure.

5. **Audit evidence**: Absence of `TENANT_PROVISIONED` and presence of `TENANT_PROVISION_FAILED` is the reliable rollback signal.
