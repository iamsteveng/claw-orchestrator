<learnings>
**Mock health-check latency collapses the startup lock window**: With an instant health mock, the control-plane completes the full acquire→dc.start→healthPoll→ACTIVE→release cycle in microseconds. The second concurrent /start request arrives AFTER the lock is released, allowing it to re-acquire and call dc.start() a second time. This is not a code bug — it's a timing artifact of zero-latency mocks.

**Fix**: Add a `healthCheckDelayMs` flag (e.g., 300ms) to the mock health check response. This holds the startup lock long enough for both concurrent /start requests to collide at the UNIQUE constraint. Reset to 0 after the lock test so cleanup is not slowed.

**The exact race**: Both relay coroutines get `status=STOPPED` from provision. Both call /start. A acquires the lock, calls dc.start(), health mock responds instantly, ACTIVE is set, lock released. B's /start arrives AFTER lock release, reads STOPPED (stale), finds no existing lock row, acquires it, calls dc.start() again. Health delay breaks this by making A's lock persist for 300ms.

**startup_locks monitor pattern**: Spawn an async background loop polling `prisma.startupLock.count()` every 50ms while the concurrent test runs. Track max observed count, assert ≤ 1 after completion.
